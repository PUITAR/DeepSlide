In today's AI-driven world, we often work with vector embeddings - numerical representations of words, sentences, and knowledge. These vectors capture meaning and relationships between concepts. When AI systems need to search for information, they must find vectors most similar to a query vector. This fundamental problem is called Nearest Neighbor Search or its approximate version, ANNS. For example, if an AI wants to find materials about Covid-19, it converts "Covid-19" into a vector and searches for similar vectors in the dataset. We measure similarity using Euclidean distance - smaller distance means higher similarity.<next>
Nearest Neighbor Search aims to find vectors with the smallest distance to a given query. This has applications in information retrieval, pattern recognition, and retrieval augmented generation systems. However, finding exact nearest neighbors is computationally expensive in practice. That's why researchers developed Approximate Nearest Neighbor Search - ANNS finds vectors with the smallest distance while mildly relaxing accuracy constraints for efficiency. There are four main categories of ANNS methods: tree-based, hashing-based, quantization-based, and graph-based approaches. Among these, graph-based approaches stand out with high answer quality and low latency, which is why we focus on them.<next>
Graph-based ANNS methods use proximity graphs, which connect similar vectors as edges. The search process starts from an entry point and traverses to neighboring nodes. The key insight comes from the "Six Degrees of Separation" theorem - any two nodes in a graph are connected by a short path. There are four main types of proximity graphs: Delaunay Graph guarantees exact nearest neighbor search but has high degree; Relative Neighborhood Graph is a subgraph of DG; K-Nearest Neighbor Graph connects each node to its k nearest neighbors; and Minimum Spanning Tree Graph is a subgraph of RNG. The search complexity in these graphs is dominated by distance computations, which is where performance bottlenecks occur.<next>
Traditional graph-based ANNS approaches face several limitations. The performance bottleneck comes from distance computations dominating search time. Memory access patterns are random, reducing cache efficiency. Graph size grows with dataset size, creating scalability issues. There's a constant trade-off between search speed and accuracy that traditional methods struggle to balance. This creates a clear need for innovation - specifically, partitioning strategies that can improve both speed and accuracy simultaneously. Our contribution addresses these limitations through a novel partitioning approach that optimizes graph structure for faster, more accurate approximate nearest neighbor search.<next>
Today I'll introduce our CSPG method for approximate nearest neighbor search. Our goal is to design a graph-based approach that achieves both high speed and high accuracy. The core idea is to randomly partition the dataset into multiple groups that share common routing vectors. This allows us to build sparser, smaller graphs for each group, enabling larger steps when we're far from the query for speed, and smaller steps when we're close for accuracy. The method consists of two key algorithms: a partitioning algorithm and a two-stage search algorithm. Let me walk you through each component.<next>
The partitioning algorithm begins by randomly dividing the dataset into multiple groups. Each group shares some common vectors called routing vectors, which are shown in red in our diagrams. For each group, we build a separate proximity graph. In our example, we have two groups: $\mathcal{G}_1$ on the left and $\mathcal{G}_2$ on the right. The key benefit of this approach is that each subgraph is sparser than a single graph built on the entire dataset. This sparsity allows for larger steps when searching, which speeds up the initial approach to the query. The routing vectors serve as bridges between groups, maintaining connectivity while enabling efficient fine-grained search later.<next>
Our search algorithm employs a two-stage approach. In the first stage, called fast approaching, we use only one proximity graph, typically $\mathcal{G}_1$, to quickly approach the query vector. This stage uses a greedy beam search with a smaller candidate set size $ef_1$, allowing minimal neighbor expansions for maximum speed. The goal is to quickly zone in on the query's nearby region. In the second stage, precise search, we carefully search around the query by considering all groups to find the final closest vectors. This stage uses a larger candidate set size $ef_2$, which is greater than $ef_1$. Unlike traditional beam search that maintains a fixed-size candidate set, our method adapts the search strategy based on proximity to the query.<next>
Let me elaborate on the first stage. The search begins from group $\mathcal{G}_1$ using a greedy beam search, which we visualize with yellow arrows in our diagrams. The objective is to quickly reach the query's nearby region with minimal expansions. This stage uses a shorter search length and fewer neighbor expansions compared to traditional approaches. After this stage, our candidate set contains the closest vector delivered from the first partition relative to the query vector. Visually, the algorithm zones into a dot parallelogram region around the query. The key insight here aligns with our overall philosophy: when we're far from the query, we use larger steps for speed, and when we're close, we'll use smaller steps for accuracy in the next stage.<next>
The second stage is where our method really shines with its cross-partition expansion technique. After the first stage, we continue the greedy beam search but with a larger candidate set size $ef_2$. The critical innovation here is what happens when we expand a neighbor that happens to be a routing vector. When we encounter a routing vector $u$, we don't just consider that single instance. Instead, we push all instances of $u$ from all partitions into our candidate set. This allows the search process to dynamically traverse different proximity graphs, effectively maximizing our search space to include as many potential closest vectors as possible. Visually, the search across multiple graphs appears as though it's conducted within a single proximity graph. With appropriate partitioning, this approach allows CSPG to outperform traditional proximity graph algorithms.<next>
