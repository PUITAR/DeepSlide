# 相关工作调研与可努力方向

## 1. 行业现状

- 给文档 / 论文自动生成一套 slides（大致合理）  
  → 卷爆：学术界几十篇，工业界一堆产品。  
- 简单的人机协作（AI 给草稿，人类修改）  
  → 在 notebook 场景尤其成熟。  
- 问题：现在的自动生成 PPT 用起来效果仍然不好…

参考链接：<https://auto-slides.github.io/>  
（本质：概括总结 + 搬图表的机器）

**Reason：**

1. AI 制作时只考虑了文章，没考虑受众层次（需不需要补充相关知识？）  
2. 抓不住重点，内容分配不合理（内容均匀分配，无法指定侧重点）  
3. 公式、表格、图像处于照搬模式，解释理解甚少（搬运为主）

## 2. 可努力方向

### 2.1 受众与时长

做 PPT 需要考虑受众，更需要考虑演讲时长。

常见要求示例：

- 为有 CS 背景的同学，在某节研究生课上讲解某篇文章，控制时长为 15 min。  
- 为无医学背景的相关从业人员，在学术交流会上讲解某篇文档，控制时长为 20 min。

**现有工作：**

- **Persona-aware D2S（EACL 2024）**  
  - 开始关注时长与用户背景（几乎唯一）。  
  - 变量仅为 `short/long`、`expert/non-expert`。  
  - 粗糙点：概念模糊（没有做明确限制），而真实用户受众与时长是很具体的。
- **PASS**：在 DOC→slides+speech 的 pipeline 里，对标题生成做了一点 audience-aware 调整。  
- **NB2Slides**：允许用户指定 audience background。  
- **普遍做法**：  
  - 塞给 LLM 的 prompt 里让 LLM “注意一下受众”，然后就没了——全靠模型自己悟，可控性弱。

#### 期望做到的

1. 考虑到受众背景，其次要将用户背景转化为一些可控制的决策（两方面）：
   - 脱离文章外能够补充一些相关知识（作为选讲）。  
     - （目前没人做过，因为大家只把用户场景当成“叙事风格”在调，而不是知识层面补充。）  
     - 感觉很必要，因为目标是“把别人讲懂”而不是“讲懵”。
   - 在文章内决定讲的内容偏专业性，还是偏叙事性（结合具体受众背景、从业背景去讲述）。

#### 方案

**方案一：用受众信息决定补充知识**

- 用受众信息，决定是否补充课外知识：  
  - 一个决策变量 —— “补充 / 不补充相关知识”。  
  - LLM 自行决定具体补充多少、补哪些点。

**方案二：用受众信息与场景信息决定“配比”**

- 用受众信息与场景信息，决定不同内容类型的“配比”：  
  - 由 LLM 负责生成配比，但要显式地“根据受众信息与场景决定内容占比”，而不是完全靠模型自己悟。  
- 认为本身受众 / 场景可以映射成一个**显式的内容比例向量**：  
  - 好处：用户对侧重点的把握清晰且可控。

示例：

- 对 “CS 研一”：  
  - 背景 / 动机：20–30%  
  - 方法：40–50%（可以讲一些细节）  
  - 实验结果 / 图：20–30%  
  - 理论推导：只保留结论 + 主要直觉，完整证明作为选讲。

- 对 “无医学背景的行业从业者”听医学 paper：  
  - 背景 / 问题痛点：40%  
  - 方法：20–30%（只讲核心思想 / 流程图，不讲太多公式）  
  - 实验结果 / 应用案例：30–40%  
  - 理论：基本不展开。

**方案三：风格**

- 语言风格：  
  - 感觉只能通过 prompt 塞给 LLM（“多讲行业话”）。  
  - 但可以明确要求其在叙述时多引入一些**行业相关的简单小例子**。

#### 限制时长

- 通过限制字数、内容长度等方式实现时长控制 —— 技术上相对容易。

### 2.2 逻辑链构建

对于同一篇论文，不同长度、不同受众的 PPT 应该**共享同一条核心叙事链**，而不是每次从头重新切块、随缘讲不同的重点。

**现有工作：**

- 现有 Doc2Slide 方法通常在**单一配置**（特定受众 + 特定长度）的前提下，为每篇文档生成一套 slide 结构：
  - 先选取内容，再按一定顺序排布，从而形成该 deck 的叙事链条。
- 当配置发生变化（例如从长讲切换到短讲、从专家切换到非专家）时：
  - 系统会重新生成一条新的 slide 序列，而不是在一条共享的叙事主干上做受控的删减和扩展；
  - 缺乏跨版本的一致性约束与评估。  
- 从逻辑链条上看，现有方法更多是在决定**哪些 part 要讲、哪些 part 不讲、它们之间的顺序关系**。  
  - 和“内容配比”那一块的区别在于：  
    - 第一块（配比）关注“某部分写多少笔墨”；  
    - 这一块关注“整体 story 结构长什么样”。  
  - 现有方法可以视为 **deck-specific** 的逻辑链。

#### 期望做到的

1. 把“演讲故事线”变成一个**显式的中间表示**：
   - 先从论文中抽取并固定一个**规范化的演讲叙事链**；  
   - 然后任何时长 / 人群版本的 PPT 都被约束为这条 storyline 的压缩或扩展视图，而不是互不相干地独立生成。  
   - 同时，这条 storyline 还可以可视化出来，让使用者选择侧重点、增删节点，从而在 **storyline-first** 的框架下生成最终的 slide deck。  
   - 也就是：为同一篇文章建立一个**可复用、跨时长 / 受众共享的规范叙事链** —— **document-specific & storyline-first**。  
   - 如果能与受众和时长进一步结合就更好了。

#### 好处

1. 演讲者有“一张脑内地图”，不会每次重建。  
2. 提升“可编辑性”：  
   - 不必和模型在“每一页的细节”上拉扯，而是一起讨论“这条线怎么走”。  
3. 受众与时长的落脚点：  
   - 在这条逻辑链上做内容配比与布置。

### 2.3 公式、表格、图像理解

> （感觉现有很多工作应该是“考虑到了”，但是实现技术难度大，所以做得不好，而不是完全没考虑。）

如果只做“文档 → PPT 的搬运”，公式、表格、图像往往只是被原样贴上去，而没有真正处理“**怎么讲**”的问题。

**现有工作：**

- PPT 相关工作大多只做“识别和搬运”，不做“可讲解化”：  
  - 把论文里的图、表、公式放到 slides 里，并不等于“帮演讲者把它讲清楚”。  
- 图像 / 图表方向有单独的工作做“把图表变成一句话 / 一段话总结”，  
  - 但**没人把这些结果系统地整合回一整套 PPT 的讲解流程中**。

#### 期望做到的

1. **公式：**
   - 直觉解释（这个式子在表达什么想法）。  
   - 符号级解释（各个符号分别代表什么）。  
   - 拆分步骤推导（2–4 步分解，让每步都容易讲清楚）。

2. **图表：**
   - 给出一段完整的说明话术：  
     - 明确横纵轴含义、不同曲线/柱子分别是什么。  
   - 给出 caption。  
   - 给出对该图的关键分析（结论 / takeaway），而不是只“贴图不讲清楚图在说什么”。
